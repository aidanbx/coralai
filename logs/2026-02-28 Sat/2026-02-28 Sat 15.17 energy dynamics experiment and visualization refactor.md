## What Was Built

### `experiments/energy_dynamics/` (new experiment)

A standalone experiment demonstrating pure energy-flow physics — no organisms, no NEAT, no neural networks. Purpose: Step 1 of the experiment ladder: establish that physics self-organizes into interesting spatial patterns before adding biological complexity.

**Files created:**
- `experiments/energy_dynamics/__init__.py` — empty package marker
- `experiments/energy_dynamics/physics.py` — all kernels and initializers
- `experiments/energy_dynamics/run.py` — GUI entry point with `EnergyVis` subclass

**Channel layout:**
```
energy  (ti.f32) — free energy at each cell
infra   (ti.f32) — infrastructure landscape
source  (ti.f32) — per-cell energy injection rate
```

**Five physics modes** (switchable live in GUI):
- `infra_attract` — energy flows toward high-infra neighbors (proportional)
- `infra_attract_cap` — infra attract + overflow spill when energy > max_energy
- `diffuse` — Laplacian equalization, ignores infra
- `infra_repel` — energy flows toward low-infra neighbors (inverse proportion)
- `infra_decay` — infra attract + cap, but infra decays each step

**Six environments** (switchable live in GUI, resets substrate):
- `patch` — source patches at corners + center, flat low infra
- `gradient` — uniform source, diagonal infra gradient
- `ring` — concentric infra peak at center, source strongest at edges
- `rivers` — diagonal sinusoidal infra channels, peak-aligned sources
- `volcano` — center source, ring of infra surrounding it
- `random` — smooth sinusoidal noise for both infra and source

Usage:
```bash
python experiments/energy_dynamics/run.py
python experiments/energy_dynamics/run.py --env gradient --shape 300
python experiments/energy_dynamics/run.py --no-gui --steps 200
```

## Visualization Base Class Refactor (`coralai/visualization.py`)

The energy dynamics experiment exposed a need: interactive normalization and per-channel view controls that should be available to **every** experiment, not just energy dynamics.

### New parameters
- `panel_width: int = 0` — widens the window by this many pixels; right strip rendered dark so sub_windows don't overlap simulation. Access `self.panel_x` / `self.panel_wfrac` for positioning.

### Normalization modes (`self.norm_mode`)
| Mode | Formula | Use case |
|------|---------|----------|
| 0 Max-norm | `v / max` | Default; relative scale |
| 1 Reinhard | `v / (v + midpoint)` | **Recommended**; painting doesn't darken everything else |
| 2 Percentile | `min(v / p99, 1.0)` | Handles sparse bright outliers |
| 3 Log | `log(v+1) / log(max+2)` | Compresses large dynamic range |

The "everything darkens when you paint" problem is solved by switching from max-norm (mode 0) to Reinhard (mode 1).

### View modes (`self.view_mode`)
| Mode | Display |
|------|---------|
| 0 | RGB blend (ch0→R, ch1→G, ch2→B) |
| 1 | ch0 only (red) |
| 2 | ch1 only (green) |
| 3 | ch2 only (blue) |
| 4 | ch0 + ch1 (no blue) |

### API changes
- `update()` split into `render()` + `show()` so subclasses can insert sub_windows between them
- `_compute_render_params()` — returns `(scale_vals, mask)` tensors for current norm/view mode
- `write_to_renderer` kernel updated: accepts `scale_vals`, `chinds`, `mask`, `norm_mode`, `sim_w`
- `_norm_view_window(sub_w)` — content method for norm + view controls; call it from any sub_window
- `render_opt_window()` — now creates a single "Controls" panel spanning the full height (simpler for standalone use; subclasses override completely)

### Subclassing pattern
Subclasses that need custom layouts override `render_opt_window()` completely (no `super()` call) and arrange their own sub_windows using `self.panel_x`, `self.panel_wfrac`, and the content methods:

```python
def render_opt_window(self):
    with self.gui.sub_window("Display", self.panel_x, 0.01, self.panel_wfrac, 0.34) as sw:
        self._norm_view_window(sw)      # norm + view buttons from base class
    with self.gui.sub_window("My Controls", self.panel_x, 0.36, self.panel_wfrac, 0.38) as sw:
        # experiment-specific controls
        ...
```

## GUI Layout: Three Non-Overlapping Panels

The previous layout (base class "View & Norm" at y=0.01–0.47, "Options" at y=0.48–0.99, then EnergyVis adding Physics at y=0.52 and Environment at y=0.78) caused panel overlap because the "Options" panel consumed the entire remaining height.

**Fixed layout for EnergyVis:**

| Panel | y start | height | y end | Contents |
|-------|---------|--------|-------|----------|
| Display | 0.01 | 0.34 | 0.35 | Norm mode (4 buttons + slider) + view mode (5 buttons) |
| Physics & Controls | 0.36 | 0.38 | 0.74 | 5 physics buttons + inject rate + max energy + paint controls + pause |
| Environment | 0.75 | 0.24 | 0.99 | 6 env buttons + Reset + Clear energy |

No overlaps. 0.01 gap between each panel.

## Key Design Notes

**`_pending_env` pattern**: Environment changes are queued in `_pending_env` by the GUI, then applied at the top of the next physics step. This avoids reinitializing the substrate mid-step.

**`sys.path` for local imports**: `run.py` adds its own directory to `sys.path[0]` so `from physics import ...` works when run as `python experiments/energy_dynamics/run.py`. The `coralai` package is imported via its installed path.

**No infra in "patch" env**: Infra is `0.1` everywhere (flat). Under max-norm this renders as solid green (normalized to 1.0). This is correct — there's just no spatial variation. Switch to infra_attract to see energy concentrate at the uniform infra floor.

**Painting interaction**: Shift+LMB drag to paint. Cursor x is remapped from full-window fraction to simulation fraction (dividing by `sim_w / total_w`) to correct for the panel strip.
