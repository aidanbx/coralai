# Ideal Repository Structure

*CoralAI repo design — 2026-02-26*

---

## What's actually wrong with the current layout

Three distinct problems, not one:

**1. The root dir is a dumping ground.**
Seven runner files at the root (`coral_runner_space.py`, `coral_runner_old.py`, `coral_runner_neat.py`, `coral_runner.py`, `minimal_runner.py`, `nca_runner.py`, `xor_runner.py`), three benchmark files, `headless_repl.py`, `test.py`, `imgui.ini`, `coralai.egg-info/`. A newcomer opening this repo has no idea where to look first. The *active* runner (`coral_runner_space.py`) is buried among six dead ones at the same visual priority.

**2. `instances/` is the wrong word for the most important concept.**
"Instances" suggests copy-of-a-class. What these actually are is *experiments* — a self-contained physics definition + NEAT config + runner that constitutes one research question. The user creates experiments. The user runs experiments. The word should be `experiments/`. And because experiments are what users *do*, they should be at the top level — not nested inside the Python package.

**3. The subdirectory grouping (`substrate/`, `evolution/`) adds indirection for no gain.**
`coralai/substrate/` has 5 files. `coralai/evolution/` has 9 files (mostly legacy). The subfolders exist because someone anticipated many files — but the anticipation was wrong. You now have `from coralai.substrate.substrate import Substrate` instead of `from coralai import Substrate`. Each extra level of nesting makes imports uglier and file navigation more annoying.

## The key changes and why each one matters

### 1. Kill the root-level runners — move to `experiments/coral/run.py`

Before: `python coral_runner_space.py`
After: `python experiments/coral/run.py`

The second form is unambiguous. There's exactly one `run.py` per experiment. Adding `--headless`, `--steps`, `--shape` flags to `run.py` replaces the current proliferation of `profile_coral_space_gui.py` and `benchmark_coral_space.py` as well — they become `run.py --profile` and `run.py --benchmark`. The benchmarking is now part of the experiment, not a separate file at the root.

### 2. `instances/` → `experiments/` at the top level

Currently a user wanting to create a new experiment must:
- Create `coralai/instances/myexp/`
- Create a separate runner at the root
- Wire them together manually

After the rename and move:
- Copy `experiments/_template/` to `experiments/myexp/`
- Edit `run.py`, `physics.py`, `neat.config`
- Run `python experiments/myexp/run.py`

That's it. The runner *is* the experiment. No separate wiring.

The other benefit: `experiments/` being at the top level signals its importance. It's where you go first, not something buried inside the Python package.

### 3. Flatten `coralai/substrate/` and `coralai/evolution/`

`coralai/substrate/` → files move directly into `coralai/`:
- `substrate/substrate.py` → `substrate.py`
- `substrate/channel.py` → `channel.py`
- `substrate/nn_lib.py` → `nn_lib.py`
- `substrate/substrate_index.py` → `substrate_index.py`
- `substrate/visualization.py` → `visualization.py`

`coralai/evolution/` → keep only active files in `coralai/`:
- `evolution/space_evolver.py` → `evolver.py`
- Everything else → `archive/` (ecosystem.py, neat_evolver.py, etc. are all broken stubs)

Result: `from coralai import Substrate, SpaceEvolver` instead of `from coralai.substrate.substrate import Substrate`.

The `utils/` subdir stays because `ti_struct_factory.py` is genuinely a utility — it's not core to the package API.

### 4. `analysis/` at the top level

Analysis tools don't belong inside the Python package because they're not part of the running simulation — they're post-hoc tools for understanding saved runs. Putting them at `analysis/` makes them a peer to `experiments/`, which is the right relationship: experiments produce data, analysis reads it.

The files ported from the analysis branch go here directly.

### 5. `runs/` as the canonical output directory (gitignored)

Currently there's both `history/` and `runs/` at the root — artifacts from different eras. The `ExperimentRunner` should always write to `runs/<experiment_name>_<timestamp>/`. This directory is gitignored. Researchers mount it to shared storage or cloud storage separately.

The key insight from the revival intentions log: **store the physics file alongside the data**. Copy `physics.py` into the run directory at start. A run is then fully self-contained — you can replay it five years later even if the codebase has changed completely.

### 6. `pyproject.toml` replaces three files

Currently: `setup.py`, `requirements.txt`, `environment.yml` — three different ways of declaring the same dependency information, with different subsets of truth. Replace with one `pyproject.toml` that covers everything. The conda environment is specified once in `pyproject.toml` and auto-generated from it.

---

## What *doesn't* change

**The Python package name stays `coralai`.** Yes, you have `coralai/coralai/`. That looks weird but it's completely standard — Flask has `flask/flask/`, Django has `django/django/`. The outer dir is the git repo name, the inner dir is the Python package. This convention exists for good reason (you can run tests against the installed package, not the source dir). Don't change it.

**The `coralai/utils/` subdirectory stays.** `ti_struct_factory.py` is a genuine utility that's not part of the public API. One level of nesting is fine here.

**`logs/` and `archive/` stay at the root.** These aren't Python and shouldn't be inside the package. They're fine where they are.

**`Makefile` stays.** The make targets (`run`, `profile`, `test`, `install`) are a good interface for the common operations. Extend them rather than replace them.

## What the root looks like after

```
coralai/
├── README.md
├── AGENTS.md
├── pyproject.toml
├── Makefile
├── coralai/
├── experiments/
├── analysis/
├── logs/
├── runs/          ← gitignored
└── archive/
```

Eight entries. Every one is immediately legible. A stranger opening the repo knows exactly where to go: `experiments/coral/run.py` to run, `analysis/` to analyze, `coralai/` if they're modifying the engine, `logs/` if they want context.

## Order of operations

Don't do this all at once — git history becomes hard to follow and imports break across multiple files simultaneously.

1. **Archive the dead runners** first — zero functional impact, immediate visual improvement
2. **Flatten `substrate/` and `evolution/`** — update all imports (a one-pass `sed` job), delete subdirs
3. **Rename `instances/` → `experiments/` and move to root** — update imports in `run.py`, `Makefile`
4. **Create `experiments/_template/`** — documents the contract for new experiments
5. **Create `analysis/` and port complexity/population files** from analysis branch
6. **Consolidate to `pyproject.toml`** — last, since it's administrative not structural

Each step is independently testable: after each one, `python experiments/coral/run.py --steps 10 --no-gui` should still work.

---

## The single most impactful change

If you do nothing else: **archive the dead runner files and move `coral_runner_space.py` to `experiments/coral/run.py`**.

That alone makes the root directory go from "what is this" to "oh, I run experiments here." Everything else is incremental improvement. The `instances/` rename is aesthetically important but functionally minor. The flattening of subdirs is nice but only affects developers. The root-level clutter is what a newcomer sees first.
