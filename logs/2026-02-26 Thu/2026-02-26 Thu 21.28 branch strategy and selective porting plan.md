## Why Not Merge

The `refactor` branch did a significant reorganization — flat module layout (`coralai/evolver.py`, `coralai/physics.py`, etc.) vs. the current nested layout (`coralai/evolution/space_evolver.py`, `coralai/instances/coral/coral_physics.py`). The `experiment/` directory structure at repo root conflicts with how `master` works. A merge would produce conflicts in almost every file. The branches were working toward a different API that was never completed.

More importantly: the branches contain *ideas*, not finished products. The Population class in `refactor` has working fixed-size genome slot management; the Experiment/Mutator abstractions are stubs. The analysis branch has working multiscale complexity math; the phylogeny system is a dataclass with `pass`. Taking only what works is the right call.

## What's Actually Worth Porting

### From `origin/refactor`

**1. Proper checkpointing (HIGH VALUE)**

The refactor branch has a working `report_if_necessary()` that actually saves state — vs. `master`'s `save_checkpoint()` which is just `pass`. The logic:

```python
# From refactor/coralai/evolver.py (report_if_necessary)
if timestep % 300 == 0:
    timestep_dir = os.path.join(self.checkpoint_dir, f"step_{timestep}")
    os.makedirs(timestep_dir, exist_ok=True)
    self.substrate.save_mem_to_pt(filepath=os.path.join(timestep_dir, "sub_mem"))
    self.save_genomes(self.genomes, os.path.join(timestep_dir, "genomes"))
    with open(os.path.join(timestep_dir, "ages"), 'wb') as f:
        pickle.dump(self.ages, f)
```

This saves: substrate memory tensor (`torch.save`), genome list (`pickle`), ages (`pickle`). The load path exists too. This is a direct, minimal port — just fill in the `save_checkpoint()` stub in `space_evolver.py` with this pattern. The directory naming convention (`history/space_evolver_run_{datetime}/step_{n}`) is clean.

**2. Fixed-size Population ring buffer (HIGH VALUE)**

The refactor `Population` class pre-allocates a fixed `num_genomes` slots:

```python
self.genomes = [None] * num_genomes
self.ages = [0] * num_genomes
self.free_indices = list(range(num_genomes))
self.combined_weights = torch.zeros(num_genomes, 1, n_acts, n_senses)
self.combined_biases  = torch.zeros(num_genomes, 1, n_acts, 1)
```

The weight tensor is allocated once at initialization at full capacity. New genomes reuse slots from `free_indices`; when a genome goes extinct its slot returns to `free_indices`. This eliminates unbounded memory growth and removes the arbitrary cull-to-100 behavior. The tensor never grows — no expensive `torch.stack()` on every radiation event.

This is the correct fix for the culling problem described in the energy dynamics log. It's a self-contained change to `SpaceEvolver` that doesn't require adopting the full refactor architecture.

**3. Benchmark/stress test structure**

The refactor branch has a `stress_test.py` at repo root. The current repo has `profile_coral_space_gui.py` which does similar work. Both approaches are valid; the stress test is more about finding breaking points rather than profiling, so they complement each other.

### From `origin/analysis`

**1. `space_checkpointer.py` — direct copy (HIGH VALUE)**

This is a clean NEAT `BaseReporter` subclass (modeled on `neat.Checkpointer`) that saves/restores simulation state with gzip+pickle. It's self-contained and doesn't depend on the refactor reorganization. It can be dropped into `coralai/evolution/` and wired to the evolver's NEAT reporter set with one line:

```python
evolver.add_reporter(SpaceCheckpointer(generation_interval=300, filename_prefix='history/checkpoint-'))
```

The analysis branch already has this file at `coralai/evolution/space_checkpointer.py` — the path matches what master expects.

**2. `analysis/multiscale_complexity.py` — direct copy (HIGH VALUE)**

This implements the renormalization group (RG) approach to multiscale complexity — the key analytical tool from the thesis. It's fully self-contained: Taichi kernels (`avg_neighs`, `upscale`) + NumPy + PyTorch. No dependencies on anything in the branch reorganization.

The approach: repeatedly coarse-grain the substrate spatial field by 2x, then compare fine/coarse correlations at each scale. High complexity = information persists across scales (like a fractal / critical system). Low complexity = either pure noise or pure order.

Functions:
- `calc_rg_flow_ragged(patterns, renorm_steps)` — produces the multi-resolution pyramid
- `calc_pixel_complexities_ragged(rg_flow)` — per-pixel complexity at each scale
- `calc_all_partial_complexities(patterns, renorm_steps)` — batch version returning `(renorm_steps, batch, channels)` tensor
- `calc_complexities(patterns, renorm_steps)` — single scalar complexity per batch/channel

This can go directly into `coralai/analysis/multiscale_complexity.py` on master.

**3. `analysis/population_analysis.py` — direct copy (MEDIUM VALUE)**

Species abundance distribution (SAD), genome distance matrix, KNN network construction. This is the ecological analysis toolkit. Self-contained except for `networkx` dependency. Good to have but less urgent than checkpointing and complexity metrics.

**4. Test suite structure (MEDIUM VALUE)**

The analysis branch has `coralai/tests/test_coralai_config.py`, `test_population.py`, `test_reportable_neat_config.py`. These test patterns that don't exist in master yet, but the test infrastructure (pytest-style) is worth adopting as a model when writing new tests.

## Recommended Porting Order

### Phase 1 — Make runs resumable (immediate, low risk)

1. **Port `space_checkpointer.py`** from analysis branch
   `git show origin/analysis:coralai/evolution/space_checkpointer.py > coralai/evolution/space_checkpointer.py`
   Then wire into `SpaceEvolver.__init__()`.

2. **Fill in `save_checkpoint()`** in `space_evolver.py`
   Use the refactor branch's pattern: `torch.save(substrate.mem)` + `pickle.dump(genomes)` + `pickle.dump(ages)`.
   Add a corresponding `load_checkpoint(folderpath)`.

3. **Add basic CSV logging** per step: total energy, total infra, cell counts per genome, mutation events. This costs one CSV write per step and immediately makes runs analyzable.

### Phase 2 — Fix the culling problem (medium effort)

4. **Adopt fixed-size genome pool** from refactor's `Population` pattern.
   Pre-allocate `combined_weights` and `combined_biases` at full capacity (e.g., 256 slots).
   Track `free_indices`. On genome death → slot returns to free list. On radiation → pull from free list.
   This is a contained change to `SpaceEvolver`; nothing else needs to change.

### Phase 3 — Add the analysis layer (medium effort)

5. **Copy `multiscale_complexity.py`** to `coralai/analysis/`.
   Wire it to run every N steps on a snapshot of `substrate.mem`, log the complexity scalar to CSV.

6. **Copy `population_analysis.py`** to `coralai/analysis/`.
   Run genome distance matrix and SAD analysis at checkpoints (expensive, don't do every step).

### Phase 4 — Physics improvements (higher effort, separate task)

These aren't from the branches but were identified as the most impactful improvements:

## How to Work Across Branches with AI

**The right workflow with an AI assistant:**

1. **Never check out the other branches** into your working directory. Use `git show branch:path` to read files from other branches without leaving master. This keeps your working state clean and avoids merge conflicts.

2. **Read first, then port manually.** Ask the AI to read a specific file from another branch (`git show origin/refactor:coralai/evolver.py | head -200`) and explain it. Then write the equivalent code for master from scratch, adapting to master's conventions. Don't copy entire files blindly — the import paths and class dependencies often won't match.

3. **One file at a time.** Port one thing, test it works (even just `python -c "import coralai; ..."`), then move to the next. This avoids the "half-ported codebase" state where nothing runs.

4. **Use git worktrees for comparison.** If you need to look at two versions side by side:
   ```bash
   git worktree add /tmp/coralai-refactor origin/refactor
   git worktree add /tmp/coralai-analysis origin/analysis
   ```
   Now you have three live directories. Read from them, work in the main one. Clean up with `git worktree remove /tmp/coralai-refactor` when done.

5. **Prefer git show for AI context.** When asking the AI to help port something, paste the output of `git show origin/branch:file.py` directly into the conversation rather than having the AI read the file. This avoids the AI getting confused about which branch it's looking at.

## What the Branches Tell Us About Intent

Reading the commit history and code together, the trajectory was:

- **master (2024)**: Working simulation, monolithic runner, minimal abstraction
- **refactor**: Attempt to modularize into testable components — stalled because the abstraction was over-designed for the current state of the project
- **analysis**: Attempt to add measurement and visualization tools — stalled because it required working checkpoints first
- **hexmap**: Geometric exploration — exploratory, stalled

The gap between refactor and analysis suggests the project hit a **chicken-and-egg problem**: you need checkpoints to do longitudinal analysis, and you need analysis to know if the system is doing anything interesting, but checkpointing requires a more organized architecture. The refactor tried to solve the architecture, got complicated, and stalled.

The right order is: **checkpointing first, analysis second, architecture refactor third** (or never, if the current architecture is adequate). This is the opposite order from what the branches attempted.
